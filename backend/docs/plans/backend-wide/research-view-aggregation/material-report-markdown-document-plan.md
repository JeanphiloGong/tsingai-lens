# Material Report Markdown Document Plan

## Summary

Collection material pages should move from card-first structured rendering to a
Markdown-first report reader. The primary material-page artifact should be a
natural-language, evidence-cited material report document, not a wall of
sample-derived cards.

The current `MaterialReportPackage` work proves that the backend can assemble
material evidence from objective-first rows, sample matrices, paper
contributions, and source references. The remaining product gap is that
structure alone still reads like a data browser. Materials papers vary too much
for a fixed card grammar to express every useful conclusion. A report document
can explain the scientific chain in prose while still preserving source
traceability through citation IDs.

This plan records the target direction for #149.

## Target Product Shape

The material detail page should read like a scientific report:

- title and executive summary
- material scope and literature coverage
- paper contribution map
- key findings
- representative material states
- thematic analysis such as densification, mechanical response, texture,
  corrosion, fatigue, and incomplete chains
- limitations and comparability boundaries
- evidence appendix with the full sample matrix and source-table traceback

The page may still contain tables, chips, sidebars, and evidence drawers, but
those are supporting controls around the report. They are not the main product
surface.

## Artifact Shape

The primary artifact should be a `MaterialReportDocument`, stored inside or
alongside the material research-view profile.

Minimum shape:

```json
{
  "schema_version": "material_report_document.v1",
  "status": "ready",
  "title": "316L Stainless Steel Material Report",
  "markdown": "... evidence-cited report markdown ...",
  "citations": {
    "E001": {
      "document_id": "doc_x",
      "source_kind": "table",
      "locator": {
        "page": 3,
        "source_ref": "tbl_doc_x_2_table_2"
      },
      "traceability_status": "direct"
    }
  },
  "outline": [],
  "warnings": [],
  "evidence_appendix": {}
}
```

The Markdown is the reader-facing report. The citation map is the machine
contract that lets the frontend turn references such as `[E001]` into clickable
source links.

## Generation Flow

The report document should be generated by a staged backend flow:

```text
objective evidence / sample matrix / source refs
  -> material evidence pack builder
  -> material report outline and theme routing
  -> section-specific material evidence packets
  -> sectioned LLM material report writer
  -> Markdown assembler
  -> material grounding validator
  -> persisted MaterialReportDocument
  -> frontend Markdown report reader
```

This is the same architecture pattern as objective report generation:

```text
scoped evidence records
  -> deterministic evidence packet construction
  -> section-scoped LLM writing
  -> Markdown assembly
  -> grounding validation
  -> persisted report artifact
```

The shared part is the method, not the concrete section schema. Objective
reports are scoped to one research objective and can use objective sections
such as objective summary, controlled comparison, mechanism chain, source
traceback, and limitations. Material reports are scoped to one material profile
across a collection and need material-specific sections such as material scope,
literature coverage, key findings, representative states, thematic analysis,
limitations, and evidence appendix.

The first material implementation should therefore build the material pipeline
directly in the research-view aggregation owner. It should not reuse the
objective report's section list, objective evidence packet builder, objective
verifier rules, or `ObjectiveReportArtifact`. If helper code becomes truly
shared after both implementations are stable, it can be extracted later as a
small internal utility. The initial implementation should keep ownership and
review boundaries explicit.

### Evidence Pack Builder

This stage is deterministic. It converts material research-view data into a
writing-ready evidence pack. It should not ask the model to rediscover facts
from raw PDFs.

Inputs:

- objective evidence units
- material sample matrix rows
- paper contributions
- source references
- source table groups
- evidence excerpts when available
- known warnings, unresolved fields, and comparability boundaries

Responsibilities:

- group evidence by paper, table, sample, property, and theme
- preserve every `evidence_ref_id` and source locator
- reduce noisy full matrices into compact report context
- include the full matrix as appendix material rather than the main report body
- name known gaps, missing joins, weak comparability, and unsupported chains

### Material Report Outline And Theme Routing

The material outline is deterministic or near-deterministic. It decides which
sections the report needs before any prose generation runs.

The first outline should cover:

- executive summary
- material scope and literature coverage
- paper contribution map
- key findings
- representative material states
- thematic analysis grouped by observed evidence families
- limitations and comparability boundaries
- evidence appendix

Theme routing should decide which evidence belongs to sections such as
densification, mechanical response, texture or model prediction, corrosion,
fatigue, and incomplete chains. A theme should appear only when the material
evidence pack contains enough evidence to support it. Missing themes should be
reported as coverage gaps instead of filled with generic prose.

### Section-Specific Material Evidence Packets

Each material report section should receive a bounded packet rather than the
whole material profile.

Examples:

- `executive_summary` receives material scope, paper coverage, key findings,
  representative states, major warnings, and high-value citations.
- `paper_contribution_map` receives paper roles, covered properties, and
  citation IDs for each paper's current-work evidence.
- `representative_states` receives selected sample or state chains, measured
  properties, process context, comparability notes, and citation IDs.
- `thematic_analysis` receives only the evidence for the selected theme and
  the warnings that constrain that theme.
- `evidence_appendix` receives the full sample matrix or a paginated reference
  to it, plus source table groups and citation mappings.

This keeps local report models from seeing a noisy 80-row or larger matrix as
the main writing context while still preserving the full matrix for audit.

### LLM Material Report Writer

This stage should use the strongest practical model configured for report
quality. It should write natural-language Markdown from the evidence pack.

The prompt should instruct the model to:

- write a scientific material report, not a generic summary
- use only evidence from the pack
- cite claims with supplied citation IDs
- explain representative material states in prose
- separate paper-local comparisons from collection-level synthesis
- mark non-comparable results and incomplete chains explicitly
- avoid global rankings when papers use different samples, processing routes,
  or test conditions

The writer should generate one section at a time from the section-specific
packet. The final Markdown document is assembled in outline order. The model
output should be Markdown plus structured metadata when needed. The Markdown is
not a convenience rendering of a card schema; it is the primary report
document.

### Grounding Validator

This stage protects traceability after generation.

It should check:

- every citation ID in Markdown exists in the citation map
- every key numeric claim is present in the evidence pack
- every key finding has at least one citation
- no unsupported global ranking is introduced
- missing evidence is represented as a limitation, uncertainty, or incomplete
  chain instead of a confident conclusion

Validation should produce warnings and can downgrade the artifact to `partial`.
It should not silently remove report text unless a safer revision pass exists.

## Frontend Rendering

The frontend should render the persisted Markdown report and enrich citations.

Expected behavior:

- render Markdown with normal report typography
- convert citation tokens such as `[E001]` into clickable evidence chips
- open source traceback from the citation map
- keep the full sample matrix and source-table appendix available below the
  report or in an evidence drawer
- avoid rebuilding scientific conclusions in Svelte components

The frontend should not call the LLM. Opening or refreshing the material page
must only read persisted artifacts.

## API And Runtime Constraints

- GET endpoints must not trigger LLM generation.
- The material report document should be generated during the build pipeline or
  by an explicit report-generation action.
- LLM prompts should receive evidence packs, not unrestricted full-PDF text.
- Markdown citations should be stable enough for screenshots, validation, and
  source traceback tests.
- The older structured fields may remain as appendix or compatibility only
  while the material page migrates, but the product-facing report should be the
  Markdown document.

## Acceptance Criteria

- The 316L material page renders a Markdown report instead of 84 representative
  state cards.
- The report includes executive summary, material scope, paper contributions,
  key findings, representative states, thematic analysis, limitations, and
  evidence appendix sections.
- Claims about values such as relative density, yield strength, hardness,
  corrosion, fatigue, or texture prediction include clickable citations.
- The full 84-row sample matrix remains available for audit, but it is not the
  main report body.
- Unsupported cross-paper rankings are absent or explicitly marked as invalid.
- The report can explain why P001 Sample 14, P004 as-SLM states, P005 high
  density rows, and P006 texture/model cases matter without forcing them into a
  uniform card shape.

## Relationship To Existing Report Work

This plan is related to, but distinct from, material review PDF generation.
The review-report pipeline can later consume this report document or share the
same evidence pack, but the material page needs a durable in-app reading
artifact first.

This plan also aligns with objective report generation: both use persisted
Markdown artifacts, sectioned generation, and citation validation. The material
report differs because its scope is a material profile across one collection,
not a single research objective workspace.

The implementation alignment is:

- same pipeline shape: evidence packets, sectioned writing, Markdown assembly,
  validation, and persistence
- different section model: material sections are driven by material scope,
  paper coverage, representative states, property themes, and appendix needs
- different packet builder: material packets aggregate across objectives,
  papers, sample rows, source tables, and citations
- different validator: material validation must prevent unsupported
  cross-paper rankings and require citation coverage for key findings
- different artifact: material output is `MaterialReportDocument`, not
  `ObjectiveReportArtifact`

The material report may later consume objective reports as supporting summaries,
but it should not depend on generated objective Markdown as its source of truth.
Its source of truth remains objective evidence, sample matrices, paper
contributions, and source references.

## Open Questions

- Should the first generation trigger be the collection build task or an
  explicit "generate material report" action?
- Should the persisted report live in the same store as material
  research-view artifacts or in a separate report artifact store?
- Should the writer output only Markdown plus citations, or a small outline
  structure for navigation and validation?
- Which model tier should be the default for material report writing in local
  and hosted deployments?
